## 2. กระบวนการหลัก: Inbound → Sorting → Outbound

### 2.1 กระบวนการขาเข้า (Inbound)

**ขั้นตอนเมื่อรถเข้าคลัง:**

1. ตรวจใบมอบหมายงาน (ชื่อคนขับ / ทะเบียน / สายรัด 3 เส้น)
2. สแกนเปิดตู้รถ
3. นำสายพานขึ้นรถ — ส่งใบนำส่งให้พนักงาน DWS
4. พัสดุทุกชิ้นผ่านเครื่อง DWS

**เครื่อง DWS (Dimension, Weight, Scanning):** ชั่ง-วัด-สแกนและคัดแยกเขตอัตโนมัติ SPK มี 22 เครื่องทำงานพร้อมกัน

**เงื่อนไขการเข้า DWS:**

| สถานะพัสดุ | วิธีจัดการ |
|---|---|
| มีลาเบลชัด | สแกนผ่าน DWS ปกติ |
| ลาเบลโดนทับ | สแกนแมนวลด้วย barcode/QR gun |
| กระสอบไม่มีลาเบล | เข้าสาย Autopacking (ต้องมีคนควบคุม) |
| รหัส 00 (อ่านไม่ได้) | วนบน **Reverse Conveyor** — พนักงานปัดด้วยมือ → ถ้ายังระบุไม่ได้ → ท้ายราง |
| หนัก >50 kg / ผลรวม 3 ด้าน >100 cm | **Oversize** → Manual Team |
| ชิ้นเล็ก / เบาเกิน | คัดแยกด้วยคน |

**พัสดุที่ไม่รับเข้าคลัง:** ไม่มีลาเบล, เสียหายหนักในปริมาณมาก, กระสอบไม่ใช่ความรับผิดชอบของคลังนั้น

หลุดสแกนรับ = สแกนส่งออกไม่ได้

### 2.4 กระบวนการขาออก (Outbound)

1. คัดแยกพัสดุตาม line ปลายทาง
2. **สแกนออกทีละชิ้นด้วย PDA** → ขึ้นรถ
3. จัดวาง: ของหนักล่าง / เปราะบาง-ท้ายรถ / VVIP/ผลไม้ แยกโซน
4. **สแกนปิดตู้** → ปล่อยรถ

**ทำไมใช้คนไม่ใช้เครื่อง:** Volume มาพร้อมกันเยอะเกินไป เครื่องรับไม่ทัน ของตกหล่น SLA ไม่ผ่าน

### 2.5 ทีม Admin (ปล่อยรถ)

**ขั้นตอนปล่อยรถ:**
1. เช็ครอบออก + คำนวณว่าของขึ้นไปทัน
2. โทรยืนยันคนขับ (ทะเบียน, จังหวัด, เส้นทาง)
3. บิลเบิกงาน (ให้คนโหลด) + ใบนำส่ง (ให้คนขับ) → **เช็คทะเบียนตรงกัน**
4. โหลด 80% → เรียกรถคันต่อไป
5. เช็คเลขสายรัดอันสุดท้าย + ถ่ายรูปตอนปล่อยรถ

**การคำนวณตัดรอบ:** ดูปริมาณงาน + เวลาถึง + % งาน ≥ 80% / ตู้แม่เต็มก่อนลูกเสมอ

## 3. ระบบ JMS และ PDA

**PDA** = เครื่องสแกนที่ทุกฝ่ายใช้รับ-ส่งของ สร้าง timestamp ในระบบ JMS ว่าของอยู่ที่ไหน เวลาไหน ใครรับผิดชอบ

**Flow สถานะพัสดุ:**
```
สาขาต้นทาง → รับพัสดุ → สแกนบรรจุกระสอบ → สแกนนำส่ง
→ GW ต้นทาง → สแกนถึง → สแกนบรรจุกระสอบ → สแกนส่งขึ้นรถ
→ GW ปลายทาง → สแกนถึง → สแกนบรรจุกระสอบ → สแกนส่งขึ้นรถ
→ สาขาปลายทาง → สแกนถึง → สแกนเปิดกระสอบ → สแกนนำจ่าย → เซ็นรับ
```

3 สถานะหลักที่ต้องสังเกต: **พัสดุมาถึง / พัสดุอยู่ที่ / พัสดุส่งออก**

## 6. แผนก Packing (JMS)

### Flow พัสดุมีปัญหา

```
พัสดุมีปัญหา → สถานะ "ระงับ"
  ├── เสียหายทั้งหมด → รอ HQ approved → ปิดงาน
  ├── เสียหายบางส่วน → รอลูกค้า confirm
  │     ├── ลูกค้าเอา → ซ่อมแล้วส่งต่อ
  │     └── ลูกค้าไม่เอา → ตีกลับสาขาต้นทาง
  └── ยกเลิก (ไม่เสียหาย) → OOP → ปิดงาน → กลับคืนสาขา
```

**ปิดงานได้เฉพาะ:** เกิน SLA + ถูกเคลม/ชดเชยแล้ว หรือเซ็นรับพัสดุแล้ว

**Re-print Label:** สูงสุด 10–15 ครั้ง / Bluetooth printer ไม่มี log → ต้องดู CCTV

**Cancel order ปัจจุบัน ~800 ชิ้น/วัน** — สแกนส่งไม่ได้ทั้งหมด เป็น hidden workload

### รหัสประเภทพัสดุมีปัญหา

| รหัส | ความหมาย |
|---|---|
| J04 | บรรจุภัณฑ์ไม่มาตรฐาน — TikTok เท่านั้น |
| J03 | เหมือน J04 แต่ไม่ใช่ TikTok |

ลงระบบผิดประเภท → ปรับ **2,000 บาท/เลข**

**เลขโมฆะ:** พัสดุอยู่เกิน **14 วัน** → ต้องมีเลข OOP ส่งคืนเขตตัดสินใจ

### กรณีพิเศษที่พบบ่อย

| กรณี | วิธีจัดการ |
|---|---|
| ตีกลับแต่ไม่ปริ้นใบตีกลับ | ไม่มีใครรู้จนถึงขาออก (SLA ตีกลับ 8 วัน = 5+3) |

| สาขาเสียหาย >30% | ตีกลับทั้งคัน สาขา repack ใหม่เอง |

## 9. แผนก Store: SOP พัสดุไม่มี AWB (无头件)

### 3 สาเหตุหลัก

ใบนำส่งชำรุด (ของเหลวแตก/ความร้อน) / ไม่มีใบนำส่ง (หลุดระหว่างขนส่ง/ต้นทางไม่ติด) / หลุดออกจากบรรจุภัณฑ์

### หลักการสำคัญ: ค้นหา AWB ก่อนลงทะเบียนเสมอ

มี **8 วิธีค้นหา** ก่อนที่จะลงทะเบียนเป็นไม่มี AWB:

1. บันทึกข้อมูลรอบรถ (ใบมอบหมายงาน / ทะเบียนรถ)
2. เมนู JMS: ค้นหา AWB ที่เสียหาย → กรอก sorting code / บ้านเลขที่ / ชื่อ / เลขบางส่วน (ใช้ `*` แทนส่วนที่ไม่ชัด)
3. เลขตกค้าง (ดึงได้สูงสุด 500 เลข/ครั้ง)
4. เลขขึ้นต้น TH → ค้นหาผ่าน "อิงตามหมายเลขคำสั่งซื้อ"
5. Order ID TikTok → ใส่ในเมนูค้นหาได้โดยตรง
6. สแกน QR code ด้วยมือถือ
7. **AI จับคู่รูปภาพ** (อัปโหลดได้สูงสุด 200 รูป/ครั้ง)
8. จับคู่ตัวอักษร (บรรยายลักษณะสินค้า)

ถ้าค้นพบเลขได้ → ส่งต่อได้เลย ไม่ต้องลงทะเบียน

### การลงทะเบียน — ข้อมูล 11 รายการ

น้ำหนัก / ชนิดสินค้า / ขนาด (กว้าง×ยาว×สูง cm) / ชื่อสินค้า / สภาพ / สาเหตุ / จำนวน / สถานีก่อนหน้า / ทะเบียนรถ / ขั้นตอนที่พบ / Platform

**ข้อห้ามสำคัญ:**
- ห้ามชั่งน้ำหนักจากบรรจุใหม่ — ชั่งจากบรรจุเดิมเท่านั้น
- ห้ามใช้คำว่า "ไม่ทราบ" หรือ "ไม่ชัดเจน" ในชื่อสินค้า → ใช้ `(N/A)เสื้อผ้าลายการ์ตูน` แทน
- บังคับถ่ายรูปอย่างน้อย 3 รูป (ภาพรวม / สินค้าภายใน / ขณะชั่งน้ำหนัก)

### Flow สถานะหลังลงทะเบียน

```
ลงทะเบียน + เผยแพร่
  → [รอการเรียกร้อง] ← สาขาต้นทางยังไม่ยืนยัน
  → [กำลังเรียกร้อง] ← ฝ่ายลงทะเบียนต้องตรวจสอบภายใน 24 ชม.
  → [เสร็จสิ้น] → ปริ้น AWB → ส่งออก/ตีกลับ
```

DP09004 รุนแรงที่สุด และตั้งแต่ มี.ค. 68 HQ ลงร้องเรียนอัตโนมัติ

## 11. Key Insights สะสม (Patterns ข้ามทุก Log)

**Accountability by Design:** ทุกสิ่งที่ J&T ออกแบบ — PDA scan ทุก handoff, ใบสั่งงานเป็นหลักฐาน, AI Routing, บทลงโทษ DP — ล้วนสร้างหลักฐานย้อนหาได้ว่าปัญหาเกิดที่ใคร แทนที่จะให้แต่ละฝ่ายเถียงกัน

**Automation ≠ เครื่องทุกจุด:** Outbound ใช้คนเพราะ volume เกินที่เครื่องรับได้ — การเลือกใช้คนหรือเครื่องต้องดู context จริง ไม่ใช่ automation = ดีกว่าเสมอ

**Escalation Structure ชัดเจน:** ปัญหาเล็กแก้ที่หน้างาน กลาง-ให้สาขาต้นทาง confirm ใหญ่-รอ HQ อนุมัติ VIP-ให้สาขาปลายทางตัดสิน แต่ละระดับมีหน้าที่ไม่ซ้อนกัน

**Data Quality at Source = Cost of Poor Quality:** ทั้ง workflow พัสดุไม่มี AWB คือต้นทุนของคุณภาพที่ไม่ดีตั้งแต่ต้น ถ้า upstream quality ดี พัสดุกลุ่มนี้จะไม่เกิดขึ้นเลย

**Negative Incentive เป็น Default:** DP09004 (500 ครั้ง/เดือน) + automation เพิ่มเข้ามา แสดงว่า J&T ไม่พึ่ง goodwill ใช้ negative incentive บังคับ compliance — realistic สำหรับ network ขนาดใหญ่
